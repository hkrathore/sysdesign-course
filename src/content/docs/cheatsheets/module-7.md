### 13 blocks. Each = use when → key decision/trade-off → the number. Skimmable in 5 minutes.

> The reflex to unlearn from the system-design track: an analytical platform does not reason in **QPS**, it reasons in **freshness × scan-cost × volume × trust**, and the cost axis is **bytes scanned**, not requests served. OLAP is not OLTP: a **projection, not a database of record**, scan-optimized not point-optimized, on **decoupled storage/compute**. The cost lever is **data layout** (partition pruning, columnar projection, pre-aggregation), *not* caching or a bigger box. And **trust is engineered**, the signature failure is a precise, confident, **silently wrong number** nobody catches for a week.

---

## Recurring laws (every block leans on these)

- **The cost model is bytes-scanned.** Serverless ≈ `bytes scanned × $/TB` (~$5/TB); provisioned ≈ `cluster-size × hours`. A 1 TB scan is ~$5; the budget killer is a **single runaway query** (unpartitioned scan, cross join, `SELECT *` over a PB), not steady traffic. Lever = layout, not cache.
- **Decoupled storage and compute** is the defining shift: one copy on cheap object storage (**~$23/TB-mo hot → ~$1–4 tiered**), elastic ephemeral compute pointed at it, scaled and paid independently, the opposite of Hadoop's bolted-together peak-sized cluster.
- **An open table format makes a lake a lakehouse.** Iceberg/Delta/Hudi = a metadata layer of snapshots/manifests over Parquet adding ACID, snapshot isolation, schema/partition evolution, time travel, concurrent writers. A directory of Parquet is *not* a table.
- **Freshness is a priced ladder, pick the rung the requirement needs** (batch→hours, micro-batch→minutes, stream→seconds, real-time OLAP→sub-second query over seconds-fresh data). Each rung costs more; the batch-vs-stream choice is made per data product.
- **Rebuildable from retained raw.** Every table must be reproducible by re-running idempotent recompute over the retained raw (same instinct as the ad-click aggregator). This is what makes backfills, bug fixes, and schema changes a parameterized re-run, not an outage.
- **The small-files tax.** Engines want large columnar files (~128 MB–1 GB Parquet); millions of tiny files make a query 5–10× slower/pricier on per-file overhead, compact (Iceberg/Delta `OPTIMIZE`).
- **The silent-wrong-number failure** is the analytical hallucination, a renamed column, a fan-out join, a timezone flip renders a beautiful, precise, *wrong* chart. Engineer trust (tests + contracts + lineage); "the query ran without error" says nothing about correctness.
- **Pick batch-vs-stream and the engine from the freshness requirement**, don't default to streaming; most analytics tolerate hours, and streaming's windowing/state/exactly-once cost is only worth a real seconds-requirement.

## Data Platforms for System Designers
A data platform is the machine between the **cash registers** (OLTP, "this entity now," point access) and the **analyst's desk** (OLAP, "all entities over time," scans), a **continuously-rebuilt, append-mostly, scan-optimized, decoupled-storage/compute projection** of operational truth, **rebuildable** from retained raw. **Never run analytics on the OLTP store** (a full-scan aggregate starves point traffic and reads a row layout column-by-column). Design in **freshness × scan-cost × volume × trust**, not QPS. **Cost ≈ bytes scanned × $/TB; the lever is layout** (partition pruning, columnar projection, pre-aggregation), not caching, a well-partitioned table turns a $5 scan into $0.05 by reading 1%. Failure modes: stale-as-fresh, **silently wrong**, runaway cost, painful backfill, schema drift.

## OLTP vs OLAP & Columnar Storage
Two ways to file the same spreadsheet: **row store = one card per customer** (fields contiguous → cheap point access, OLTP, B-tree) vs **column store = one folder per field** (values contiguous → cheap scans, OLAP). The orders-of-magnitude win is **three compounding multipliers**: **projection** (read ~k/N columns, a single-column aggregate on 50 cols reads ~2%), **compression** (**3–10×** on homogeneous columns via dictionary/RLE/delta/bit-packing), **vectorized/SIMD** (**10–100×** per core). **Parquet/ORC** carry **min/max zone maps** → **predicate pushdown skips blocks**, tightest when **sorted/clustered** by the filter. Format = delegated bake-off (both open); **codec = CPU-vs-IO** (Snappy hot, Zstd cold/egress-billed). Row stores still win for **selective lookups + frequent updates**; **reject HTAP** unless a requirement pays for it.

## Data Warehouse, Lake & Lakehouse
Three architectures by **guarantees and economics**, not logos: **warehouse** (proprietary storage, schema-on-write, governed SQL, turnkey, locked-in, storage premium); **data lake** (open files on cheap object storage, schema-on-read, cheap/open but ungoverned → **swamp**); **lakehouse** (open files **+ a table format** = warehouse guarantees on lake economics). Under it all: **decoupled storage/compute**, one copy at **~$23/TB-mo (→~$1–4 tiered)**, elastic compute on demand. The **open table format** (Iceberg/Delta/Hudi) = a metadata layer of snapshots/manifests over Parquet (ACID, snapshot isolation, evolution, time travel, concurrent writers), the commit is one atomic snapshot swap. Choose on **scale × workload (BI vs ML) × lock-in × ops maturity**: warehouse for BI-first/ops-light, lakehouse for PB + BI&ML + openness. Raw Parquet is **never** a system of record; format pick = delegated bake-off, *using one at all* = the load-bearing call.

## Distributed Processing Engines: Spark & Flink
**Spark** = a warehouse that periodically re-sorts the whole floor: lazy DAG, narrow ops pipeline locally, **wide ops force the shuffle** (all-to-all, write-to-disk → read-over-network, a barrier, **spills on skew**, the #1 "hangs on the last task" cause). Kill it with **broadcast joins** (small side, no shuffle of the big table, often 10–100×) + pre-partitioning + map-side aggregation. **Flink** = a baggage system that never closes: per-event, **event-time + watermarks** (δ = freshness-vs-completeness dial), **RocksDB keyed state** (exceeds RAM at ~µs vs ns), **Chandy-Lamport checkpoint barriers** (consistent snapshot without stopping → restore + rewind offsets; interval = recovery-vs-overhead, typical 10–60s). **Exactly-once = atomic sink+offset commit**, not "each event once"; **backpressure** = graceful slowdown. **Pick the engine from the freshness requirement**: hourly→Spark, seconds→Structured Streaming micro-batch, sub-second stateful→Flink.

## Real-Time OLAP & Serving Engines
The **second analytical store**, the **arrivals board** (Druid/Pinot/ClickHouse: *simple aggregations, sub-second p99, hundreds-to-thousands QPS, seconds-fresh*) vs the warehouse's **research library** (*complex/flexible, low-concurrency, seconds-to-minutes*). Three numbers answer "why not the warehouse?": **concurrency, latency, freshness**, each an order tighter. Sub-second-at-scale = **columnar time-partitioned immutable segments + rollup at ingest (10–100× fewer rows) + bitmap/inverted indexes + memory-resident hot data**. Shape: **real-time nodes** ingest Kafka → hand sealed segments to **historical nodes** off deep storage (S3); a **broker scatter-gathers** both; a **batch path overwrites segments** to backfill/correct (the lambda-ish speed-vs-truth handoff). **Rollup costs raw granularity** → never roll up on high-cardinality IDs; keep raw in S3 for drill-down. Distinct from observability time-series; engine pick = delegated bake-off (often "whichever the org runs").

## Data Ingestion & CDC
Ingestion is the **EL of ELT** and where platforms bleed, design for **completeness, low source impact, rebuildability**. Three modes: **full load** (tiny dims), **query-based polling** (`updated_at`, append-only only, **misses hard deletes** and adds DB load), **change data capture** (the default for any mutable table). **Log-based CDC** reads the **WAL/binlog/redo** the database already writes for replication via Debezium → Kafka → bronze lake: captures every insert/update/**delete** at **seconds**-freshness with near-zero source impact. **ELT > ETL**: land **raw** first (cheap object storage), transform in-warehouse, stay **rebuildable**. **Buy-vs-build = arithmetic**: `MAR × rate` (Fivetran balloons at volume) vs `infra + ops` (self-host Debezium cheap per row at scale), often **hybrid**. **Idempotent loads** (merge on pk + LSN) make replays safe; watch the **replication slot** filling the source disk.

## Orchestration & Transformation
A platform is a **kitchen at dinner service**: the **expediter** (orchestrator, Airflow *task-centric* vs Dagster/Prefect *asset-aware*) calls the order, **dependency-driven firing** (fire B on A's recorded success, not a timing guess), retries+backoff, the 07:00 **SLA** (reuses the scheduler machinery; a missed run is silent unless you alarm), and the **line cook** (**dbt**) follows a versioned, tested recipe: SQL models run *in* the warehouse, `ref()`-derived DAG + lineage, built-in tests, incremental models, replacing opaque scripts/stored procs. The load-bearing property: every task **idempotent + partition-scoped** (delete-insert / `INSERT OVERWRITE`, **never blind-append** → no double-count), so **backfill = N partitions × per-partition cost** (90 × 200 GB × $5/TB ≈ $90, minutes at parallelism), a costable re-run, not an outage. **Orchestrate-the-transform**; don't transform-inside-ingestion. The seam = the `dbt build` task.

## Analytical Data Modeling
The **briefing pack, not the filing cabinet**, you **normalize (3NF) for safe writes, denormalize for cheap reads** (redundancy is free on columnar, joins across millions of rows are not). **Dimensional modeling (Kimball):** central **fact** (measures you SUM, FK-heavy) + surrounding **dimensions** (who/what/when/where you slice by) = the **star**; **declare the grain first** (what one fact row means) or you get silently-wrong double-counts. **Star over snowflake** (fewer joins; dictionary-compression makes the redundancy near-free). **SCD Type 2 over Type 1**: new row + valid_from/valid_to + **surrogate key** preserves point-in-time history; Type 1 overwrite **silently rewrites the past**. **Gold in the medallion model is the dimensional layer** (Kimball = shape, medallion = path, both rebuildable). **One Big Table** = a per-mart, columnar-enabled optimization (no joins, rigid), not the default. A **gold pre-aggregate** turns a 50M-row scan into ~thousands of rows. Disagreeing dashboards = a **semantic-layer/governance** failure (one governed metric definition), not a query bug.

## Data Quality, Testing & Contracts
The signature failure is a **confident wrong number nobody catches for a week**, trust is engineered. Decompose "good data" into **seven testable dimensions**: freshness, volume, schema, distribution/values, uniqueness, referential integrity, completeness (most breaks = schema change, fan-out, unit/timezone error, partial load). Those tests are the **regression suite for data**, the exact LLM-eval-gate analog: a fixed assertion set every run must pass before it's trusted; run in CI, fold every incident in as a permanent test. **Shift-left** (a **contract** at the producer boundary, prevents, **zero detection lag**, costs **producer buy-in**) vs **shift-right** (**observability/anomaly detection**, catches, broad coverage, damage already flowed), use **both**, contract the high-blast-radius feeds. On failure: **circuit-break** (wrong ≫ stale: finance/customer-facing/prod ML, caught before publish, at cost of false-positive stoppages) vs **alert-and-continue** (stale ≫ wrong: internal/low-stakes). **Detection lag is the scoreboard**: week → 5-min → zero. Pair with **lineage**: tests detect, lineage scopes impact + cause.

## Governance, Catalog, Lineage & Data Mesh
At scale the hardest problems are **organizational**, the **librarian** for a million-book library answering four questions: *find it* (**catalog**/discovery, what tables exist, what they mean, who owns them), *allowed to see it* (**column/row-level security + masking + audit**, scaled by **tag-based/ABAC policy** because hand-grants explode as datasets × roles × columns), *trace it* (**column-level lineage** for debugging a wrong metric, impact analysis on a schema change, compliance/provenance, a system, not docs), *who owns it* (ownership + **data contracts**). The **catalog is the governance chokepoint** (Unity Catalog / Glue / Polaris), **buy, don't build** it. Then the **Conway's-law** org fork: **centralized** (consistent, but the **bottleneck** past dozens of domains) vs **data mesh** (domain-owned products + self-serve platform + federated computational governance, scales but risks inconsistency/duplication and is **cargo-culted at small scale**). Decide on **domain count + proven bottleneck**, not hype; the realistic large-org answer is a **hybrid** (central paved road + federated policy + domain products), with **contracts** as the inter-domain API.

## Data Platform Economics & FinOps
A **self-storage facility with a per-visit forklift fee**, **two decoupled bills**. **Storage** = `bytes × $/TB-month` (paid idle; cut by **retention, tiering hot→cold→archive ~20×, compression 3–10×**). **Compute** = `bytes scanned × $/TB` (serverless) or `cluster × hours` (provisioned), cut by **partition pruning** (scan ~1% → $5 to $0.05), **materialization** (the $15/scan × 200/day ≈ **$90k/mo** scan-on-read dashboard → tens of $, trading staleness + refresh), **auto-suspend** (stop paying idle), **spot** for interruptible batch (60–90% off). Pricing choice = the **utilization crossover** (~60–70%): on-demand for spiky/low, **reserved (30–60% off) for steady-saturated**, reserve the baseline, burst the top, **fix waste before committing**. The killer is a **single runaway query**, so guardrails (per-query byte caps, `require_partition_filter`, quotas) are **design, not afterthought**. Govern with **attribution + unit economics** (`$/dashboard`, `$/team`), chargeback creates the incentive a central bill never does. Cost is layout and policy, not hardware.

## Data Strategy & Build-vs-Buy *(leadership)*
Sourcing **power for a factory**: plug into the **grid** (managed warehouse) and pay the **meter** (consumption) first; build your own **generator** (self-hosted lakehouse + a crew) only when sustained load beats the meter. Spectrum: **managed SaaS warehouse → assembled modern data stack → self-hosted open lakehouse → build-from-scratch** (the **rejected default** for all but hyperscalers, naming and killing it is the altitude signal). Answer as **Clarify (volume/team/workload/residency/cost/lock-in) → Options → Sequence → Trade-off/reversibility**, sequence = **buy-to-move-fast → standardize on an open table format → self-host where scale earns it**. The **crossover is whether savings clear a standing platform team (~$1–2M/yr in 4–8 engineers), not the per-TB sticker**, a wash at ~$1.5M managed, self-host clearly wins only at high six/seven figures of *steady* volume. **Open table format = the optionality lever** (reversible rental); **proprietary storage + a self-hosted team = one-way doors**. 2026 calibration: rent elastic cloud (no on-prem Hadoop), open format is the integration point, buy the commodity (ELT+dbt+connectors), **consolidate** the fragmented seven-tool stack.

## Leading a Data Platform Org *(leadership)*
The Director job is the **operating model, not the pipeline**, run data like a **water utility** (guarantee the metered, pressure-checked pipes) not a **bottled-water crew** (carry buckets / fill dashboard tickets). Four decisions: **(1) Org shape** fit to size, centralized (consistent, bottleneck) → embedded (fast, inconsistent, rarely a destination) → **hub-and-spoke (the default, central platform hub + embedded analytics-engineer spokes, at a coordination cost)** → **mesh** (domain-owned products + federated governance, *only* at genuine large scale, don't cargo-cult). **(2) Data-as-a-product**: named owner, SLA, docs, support per dataset (best-effort data erodes trust, the costliest failure). **(3) Reliability contract**: freshness + completeness **SLOs**, a **data on-call rotation** (~5–8 engineers, tight alert budget) paged on breach, a "the number is wrong" runbook with a backfill/replay path. **(4) Access**: governed **self-serve** over a semantic layer scales the function; the central-build ticket queue is a **bottleneck to reject**. 2026 calibration: data-is-IT → product function, no-SLAs → SLOs+on-call+contracts, one team → federated ownership, do-everything data scientist → specialized roles (analytics engineer = keystone), + AI-assisted analytics raising the value of the governed semantic layer.

## Director through-line (all 13)
Classify the **real binding constraint** (freshness rung / scan-cost / unbounded volume / engineered trust / GPU-free but bytes-metered economics / org bottleneck), not "which warehouse." · Reason in **bytes scanned and the freshness ladder, never QPS**; the lever is **data layout** (partition pruning, columnar projection, pre-aggregation), never a cache or a bigger box. · Every decision names its **trade-off and the rejected alternative**, and **quantifies the dropped side** (bytes scanned, $/month, freshness lag, detection lag, the team-cost crossover). · **Engineer trust explicitly**, the silent wrong number is the signature failure; tests + contracts + lineage are first-class, not polish. · **Delegate the bake-off with a stated prior** ("the platform team benchmarks Iceberg vs Delta on our CDC-upsert and multi-engine profile; my prior is Iceberg for engine neutrality"). · Always carry the **cost, governance, and on-call** dimension, and choose the **org shape on domain count and proven bottleneck**, not hype.

6 data-platform problems. Each = opening move → pivotal decision/trade-off → the numbers. Skimmable in 5 minutes.

> **The reflex these reward:** separate **storage from compute** first, pick the store from the query's **access-pattern + freshness**, and treat **cost, trust, and governance as first-class**, not afterthoughts. Each problem maps to a foundations concept lesson; the problem is where the concept gets built end-to-end.

---

## Recurring moves (the same patterns, reused across the module)

- **Open by separating OLTP from OLAP, then ask the access-pattern + freshness question.** Every opener splits two workloads the question conflates (warehouse-vs-lakehouse, user-facing-vs-ad-hoc, exactly-once-where, centralized-vs-mesh, tests-vs-ML), and the answer flips the architecture before you draw a box.
- **Decoupled storage/compute is the decade's idea.** One cheap open copy on object storage (S3/Parquet), elastic compute pointed at it on demand, independent compute pools so a heavy ML scan never starves the finance dashboard. Lakehouse and warehouse share this foundation.
- **Scan cost = bytes scanned, controlled by layout, never by bigger clusters.** Partition (by day/time), cluster/Z-order on the filter key, pre-aggregate the hot shapes, compact small files. A bigger cluster makes the wrong-shaped query *faster and more expensive*; the fix is always "scan less."
- **Rebuildable from retained raw.** Keep bronze/raw forever-ish; everything downstream is an idempotent recompute (the rebuildability invariant). A logic bug or schema change becomes a re-run, not a panic. Never load straight to curated/gold.
- **Never serve sub-second user-facing from the batch warehouse.** It's flexible and cheap but seconds-to-minutes at low concurrency; the user-facing surface needs a separate real-time OLAP store. They're *complementary, not competing*.
- **The speed-vs-truth split, everywhere.** A fast approximate projection for the product/dashboard + a complete authoritative copy in the lake that back-corrects it. Never bill, pay, or sign a regulator's answer off the fast path.

## Data Warehouse / Lakehouse
**Crux / opening move:** "Governed-SQL analytics warehouse (BI-first, turnkey) or open lakehouse (BI **and** ML on one open copy, lowest cost, no lock-in)?" It's a *platform-architecture* problem, not database selection; separate storage from compute first.
**Pivotal decision + trade-off:** **Open lakehouse** = object storage + Parquet + an **open table format (Iceberg/Delta)** + a catalog. The table format is non-negotiable: it brings ACID commits, snapshot isolation, schema/partition evolution, and time-travel that make a *lake* queryable like a *warehouse*, so **one copy serves BI and ML**. *Rejected:* raw Parquet dirs (no ACID/evolution), proprietary warehouse storage (lock-in, dear at PB). Refine through **medallion** (bronze raw-retained → silver clean → gold marts), ELT not ETL. Managed warehouse for BI-first/ops-light; the two converge (managed engines query external Iceberg).
**The numbers:** ~10 TB/day raw → ~5–8 PB over 3 yrs; **storage = tier cold partitions** (hot ~\$23/TB-mo → archive ~\$1–4, a 5–20× cut). **Compute = bytes scanned**: a naive 2 TB un-pruned scan ≈ \$10/query, \$600k/mo; pruned (partition + cluster + project) scans ~5–20 GB ≈ single-digit-thousands/mo, a 50–400× cut. Watch the **small-files problem** (compact to ~100 MB–1 GB) and **upserts/deletes** (table-format MERGE, MoR for write-heavy CDC).

## Real-Time OLAP Serving
**Crux / opening move:** "User-facing (sub-second p99, high QPS, bounded query shapes, seconds-fresh) or internal ad-hoc (flexible arbitrary SQL, low concurrency)?" Sub-second-at-high-concurrency is an *architecture* problem, not a tuning problem on a general store.
**Pivotal decision + trade-off:** A **purpose-built real-time OLAP store** (Druid/Pinot/ClickHouse) built from **immutable, time-partitioned, columnar, bitmap-indexed, pre-aggregated (rolled-up) segments**. **Rollup at ingest** is the headline call: it makes sub-second p99 *structural*, but **costs raw granularity** and **blows up on high-cardinality dims** (use **sketches/HLL** for uniques; drill down in the lakehouse). Two tiers stitched by a **scatter-gather broker**: **real-time nodes** serve in-memory just-arrived data; sealed **immutable segments** go to **historical nodes** over deep storage, with a **metadata-driven ownership flip** so the boundary isn't double-counted. *Rejected:* the warehouse (too slow), a **Redis precompute** (rigid, only materialized slices; use it as a broker *result cache*), raw/no-rollup (bytes-scanned tax).
**The numbers:** 500k events/s ingest; **rollup compresses ~10–100× (~20× typical)**, ~8.6B rows/day → ~430M; raw ~1.7 TB/day → segments ~5–10 GB/day after rollup+columnar (~100–300× footprint cut). A query touches **tens of MB across a few segments → ~20–50 ms** at 3,000 QPS. Hot/cold tiering by age tracks the access skew. Never serve exact-for-money from here (that's the exact batch path).

## CDC Ingestion & Streaming ETL
**Crux / opening move:** "What freshness does the destination need, and **where is exactly-once mandatory vs at-least-once-plus-dedup tolerable**?", sets streaming-vs-micro-batch and how much exactly-once machinery to buy. It's a *correctness* problem wearing a *plumbing* costume; the failure mode is **silent**.
**Pivotal decision + trade-off:** **Log-based CDC (Debezium on the WAL/binlog) → Kafka (partitioned by primary key) → Flink → idempotent MERGE into bronze.** Log-based beats **query-based polling** because polling **silently misses deletes** and **loads the source**. **Effectively-once** = PK partition (per-row order) + **MERGE guarded by `source_lsn`** (replays/out-of-order = no-ops) + transactional checkpoint. The **Kafka buffer** is load-bearing: without it a stalled apply lets the WAL truncate unread changes, *permanent, silent loss*. Two silent killers: **schema drift** (registry: additive auto-flows, rename/drop **surfaced for review**, renames masquerade as additive) and the **snapshot-stream gap** (snapshot-then-stream cutover + LSN dedup). Per-row order is sacred, **can't sub-shard a hot key**; push aggregation downstream instead.
**The numbers:** ~50k change-events/s sustained, ~150k peak (~110 MB/s) → **16–32 partitions** per high-volume topic; ~10s Flink checkpoint → **seconds-to-~10s commit-to-bronze lag**; **effectively-once costs ~10–30% throughput** (paid where a double-applied payment is an incident). 5B-row backfill ≈ ~7 hrs chunked against a replica. At 4.3B changes/day, **self-host beats per-row managed pricing** (Fivetran/Airbyte five-to-six figures/mo).

## Product / Behavioral Analytics Platform
**Crux / opening move:** "Interactive self-serve (PM funnels, sub-second, bounded grammar), deep ad-hoc/ML (data scientist, arbitrary SQL, years of history), or **both**?" The track's **synthesis** problem, it *composes* the firehose + OLAP + lakehouse + event-time pieces, it doesn't invent.
**Pivotal decision + trade-off:** A **dual path from one firehose**, one store can't be both. A **real-time OLAP store** serves the *bounded grammar* (funnel/retention/segment, event-time-ordered + pre-aggregated hot funnels); a **lakehouse** is the *flexible complete* ad-hoc/ML truth and the reconciliation **authority** that back-corrects the fast projection. *Rejected:* single warehouse (too slow for sub-second self-serve), precomputed cubes (rigid, can't explore the 40 variants a PM asks). The un-glamorous middle is the correctness story: **identity stitching** (anon→user, with **batch re-resolution** to back-correct retroactive merges over retained raw) and **event-time sessionization** (server-derived, watermarks, reused from the event-time aggregation pattern). Events **schema-flexible behind a governed semantic catalog** (the conformed layer), neither rigid schema-on-write nor ungoverned `btn_click`/`button_clicked` rot.
**The numbers:** ~50M MAU × ~40 events/day ≈ **2B events/day**, ~23k/s sustained, ~70k peak (firehose sized in the CDC-ingestion problem); ~1 TB/day raw → **~75–150 TB** retained 2 yrs after compression+tiering. **Funnel scan-cost** is the headline: a 5-step/30-day funnel naively ≈ 60B events = **40s on the lakehouse → tens of ms** on the event-time-ordered OLAP store; pre-aggregating hot funnels = a lookup. Self-serve is a **bounded grammar**, not arbitrary SQL (that goes to the lake).

## Data Lake + Governance / Data Mesh
**Crux / opening move:** "How many domains/teams produce data, and **is the central data team the bottleneck** (are domains waiting weeks to publish or get access)?" An **org problem wearing an architecture diagram (Conway)**, the trap is solving only the tooling half and ignoring the org.
**Pivotal decision + trade-off:** Two interlocking decisions. (1) **Operating model:** below the crossover, a **centralized platform team** is correct and simpler; past it, a **federated data mesh**, domains own **data products** (owner + SLA + versioned **contract** + classification), a central platform team owns the **self-serve paved road** + global standards. **Don't cargo-cult mesh**, it trades a central bottleneck for distributed inconsistency, pay only when the bottleneck is real. (2) **Governance plane = a policy chokepoint, not a human queue:** **catalog** (find *and judge*: owner/freshness/quality/lineage), **policy-as-code** (tag-based ABAC, **column/row-level**, computationally enforced, scales with rule-count, not dataset-count), **column-level lineage** (trust/impact/GDPR transitive-closure, auto-captured). **Federated computational governance** resolves autonomy-vs-coherence. Mesh's real failure cause is **org maturity** (people), not tech.
**The numbers:** ~40 domains, ~2,000 datasets, ~15-engineer central team → **~12,000 engineer-days/yr of demand vs ~3,300 supply ≈ 3.6× shortfall** (the bottleneck, quantified, you can't hire past a serialization point). **Crossover ≈ <10 domains / few-hundred datasets** (below it, centralized fits). ~2,000 tables × ~30 cols = **~60,000 column-policy decisions** → forces ~tens of tag-rules, not per-object grants. Onboarding multi-week → **~2–3 days**; discovery days → **seconds**; lineage **>90%** of gold/silver.

## Data Quality & Observability Platform
**Crux / opening move:** "**Declarative tests** (explicit, deterministic, high-precision, must be authored/maintained) or **ML anomaly detection** (auto-coverage of thousands of tables, catches unknowns, false-positive + opaque)?" The answer is **both, scoped by table criticality**. A data platform that monitors data platforms; it kills the silent wrong number.
**Pivotal decision + trade-off:** **The monitoring must cost an order of magnitude less than the platform it watches**, you can't full-scan a PB to check a PB. So collect health from **metadata first** (freshness/volume/schema, *free* from `information_schema`) + **sampled aggregates** (distribution, ~MBs), with a **hard scan budget**; full scans only for rare tier-1 exact assertions. Detection split by blast radius: **ML baselines broad** over all tables (the unknowns, seasonality-aware), **declarative tests on the ~100–200 tier-1** tables (revenue/regulatory/exec, where a false negative is a board-level event). The **lineage graph** is the diagnosis layer, upstream→root cause, downstream→impact, and **dedup** (one root cause = one incident, not 50 symptoms). **Circuit-break only tier-1/opt-in/test-confirmed, quarantine-first**; **alert-and-continue by default** (a false positive must never halt the business). Reuses observability's TSDB/anomaly/alerting.
**The numbers:** ~5,000 tables (~100–200 tier-1), ~7 checks/s, low QPS, **brutal in scan cost if naive**: full profiling ≈ 250 TB/interval, 6 PB/day, **~\$900k/mo**; metadata-first + sampling lands it in **low thousands/mo, a ~500× cut**. Detection lag: a week → **minutes-to-an-hour** (100–300× cut). **Precision is a budget:** 600k checks/day × 1% = **6,000 false alerts/day** → a muted platform (the observability pager-muting failure, for data); fix with lineage dedup + seasonality baselines + tiered severity. Health TSDB ≈ ~100k series, single-digit TB (cardinality is *not* the constraint here, scan cost is).

## Director through-line (all 6)
Classify the **binding constraint** first, and on this module it's almost never QPS. It's **storage economics + scan cost**, **sub-second-at-concurrency**, **correctness/exactly-once**, **dual-path reconciliation + identity/event-time**, **org throughput / Conway**, or **monitoring-cost + precision**. · Decouple **storage from compute**; serve the user from a fast projection and keep the **complete, rebuildable, authoritative truth in the lake** that back-corrects it (the speed-vs-truth split, everywhere). · Control cost by **layout** (bytes scanned), never bigger clusters; treat **governance, lineage, trust, and the monitoring bill** as first-class design surfaces, not afterthoughts. · **Delegate IC depth with a stated prior** (the table-format bake-off, the OLAP engine, the Flink checkpoint interval, the catalog tool, the anomaly model) while keeping the architecture: decoupling, open formats, the speed-vs-truth split, scan-cost-by-layout, the operating-model call. · Carry the cost line and the on-call/compliance dimension, a Director owns the budget *and* the GDPR answer.

> **Spaced-repetition recap:** 6 data-platform problems, one spine. Always **separate OLTP/OLAP and ask access-pattern + freshness** first, then **decouple storage from compute** (one open copy on object storage, elastic pools).
